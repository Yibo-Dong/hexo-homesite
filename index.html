git

CH1

git处理信息的方式：

  • 直接记录快照，而非比较差异。并非保存每一次的新版本与原先版本的差异列表，而是创建一系列“快照”，并保存其索引。
  • 大多数操作本地执行-> 低延时
  • 保证完整性 -> 保存校验和

git中的文件状态

  • modified ： 已修改，还未保存
  • staged：已暂存，会保存在下一次提交的快照中
  • committed：已安全保存

关于配置文件

三级配置：
  • /etc/gitconfig包含系统上每一个用户和仓库的通用配置。可以通过git config --system来修改
  • ~/.gitconfig只针对当前用户，可以使用--global来修改
  • .gte/config当前仓库中的config文件，只修改当前的仓库。可以使用--local来实现，为默认值
检查配置文件：git config --list
设置别名：git config --global alias.br branch 将branch指令设置为br。

CH2

提交与修改

  • git add只是提交到暂存区。如果在提交后又做了修改，需要再提交一次。

  • git status可以查看文件的修改、新增等。使用git status -s可以简洁地显示，分为两栏，暂存区和工作区。

  • git commit会将暂存区归档。-m选项会允许把commit message放在同一行。如果想要跳过暂存，可以使用-a来提交，会自动暂存并提交所有已经被追踪过的文件

  • git rm可以从暂存区中删除文件，在下一次提交后，就不会再纳入版本管理了。如果是已经commit过的，则需要-f。如果想从暂存区移除，但仍然保留在本地（例如，写错.gitignore导致上传了一系列无意义文件），使用git rm --cached有效。

  • git mv会让git意识到一次renaming操作。

  • git log可以查看修改的日志，而-p选项会显示每一次所引入的差异。-2会限制只显示最近两次提交。如果想查看简略的统计信息，可以使用--stat选项。还有--pretty选项，可以修改展示历史的方式。甚至使用format来定制。

  • git last可以直接查看最后一次提交。

忽略文件

  • .gitignore可以配置一系列的规则。
  • 以#开头会被忽略
  • !表示取反。例如*.a忽略了所有的.a文件，而!lib.a可以要求追踪lib.a文件z
  • *表示零个或多个任意字符，**表示任意中间目录
  • /xx只会匹配当前目录下的xx，而xx则会递归地忽略所有目录下的xx

查看差异

  • git diff可以查看尚未暂存的文件中有哪些更新
  • git diff --staged查看已经暂存的文件中有哪些更新

撤销操作

  • git commit --amend可以修改commit message，或者是补交文件。
  • git reset HEAD xxx可以将暂存区中的xxx重置回HEAD（也就是上一次所提交）的状态。
  • git checkout -- xxx可以将本地对xxx的修改完全取消（恢复上一次的样子），这意味着本地修改会全部木大。

关于远端仓库

  • git remote可以查看所有的远端服务器。-v可以显示远端的git对应的url。
  • git remote add <shortname> <url>可以添加远端仓库，同时指定一个方便的简写。
  • git fetch可以拉取仓库中没有的信息。
  • git pull适用于当前分支设置了追踪远程分支的情况，会自动地拉取并合并。git clone会自动设置本地master分支跟踪远端master分支。
  • git push可以将分支推送到远端。
  • git remote show origin会查看远程仓库的URL与追踪信息。
  • git remote rename 可以修改远程仓库的简写名。

打标签

标签有两种：
  • lightweight，轻量级标签，只是一个特定提交的引用，操作方式为git tag <version>即可。
  • annotated，附注标签，是数据库中的一个完整对象，支持签名、校验等操作。操作方式为git tag -a <version> -m "message"

想要查看标签，可以使用git show。lightweight只会有提交的信息，annotated会有附注的信息和作者信息等。

同样可以给过去的分支打标签，只需要git tag <hash>就可以了。

标签并不会被默认地推送到远端，需要使用git push origin <tag>来推送特定标签, 或者使用git push origin --tags来推送所有标签。如果想要删除，git tag -d <tag>可以删除本地的标签，git push origin --delete <tag>删除远端标签。

标签可以协助检出(checkout)，但最好不要直接在某标签上修改后提交，因为这会使得新提交不属于任何分支，且无法通过除了指定哈希以外的方法访问。

CH3 分支管理

在提交的时候，git会计算每一个子目录的校验和，然后把它们保存成一个树状结构的对象集合。其中有三种对象：

  • blob对象：记录每一个文件快照
  • 树对象会记录文件结构，指向各个文件；
  • 提交对象会记录本次提交的相关信息，指向树对象。提交对象中也会记录上一次提交的对象。

git的分支说白了就是一些指针。HEAD也就是相当于一个本地分支的别名。

分支操作：
  • 创建分支：git branch xxx。默认地并不会自动切换到新分支上去。
  • 查看分支：git log --decorate可以查看当前各个分支指向的对象。git branch本身会查看当前分支的列表。-v选项还会额外查看每个分支的最后一次提交。--merged可以查看已合并的分支，--no-merged可以查看未合并的。
  • 切换分支：git checkout xxx 这样HEAD就会指向xxx了。如果需要新建一个并切换，使用-b参数。
  • 删除分支：git branch -d xxx来删除分支。
  • 合并分支：git merge xxx 把xxx分支合并入当前分支。
    • 如果其中一个沿着往前走可以到达另一个，那么合并的时候会“快进”（fast-forward）。
    • 如果在过去某个公共祖先之后分叉了，则git将进行一次“三方合并”。它将创建一个新的快照，并且自动创建一个提交指向它（于是，这个提交有不止一个父提交）。
      • 当然，这个三方合并并不一定可以自动完成。如果两个方向上对同一个地方作了不同的修改，就会需要手动消除冲突。这时候使用git status可以查看含有未解决冲突的文件。而本地修改消除冲突以后，git status会变成”all conflicts fixed but you are still merging”
远程分支
  • 使用git remote show <remote>来查看分支的详细信息。
  • 远程追踪分支：它们是远程分支的引用，以<remote>/<branch>形式命名。它记录了最后一次和远端通信时候的<branch>的状态。当其它用户修改了<branch>时，我们自己的<remote>/<branch>并不会移动。如果需要更新，只需要git fetch <remote>就可以。
  • 推送：git push origin <branch>可以把本地的<branch>推送到远端。它实际上是一个简写，意思是将本地的<branch>推送到远端的<branch>上。
  • 追踪分支：从一个远程分支检出一个本地分支会自动创建追踪（被追踪者，被称作”上游”，也即upstream），从而可以使得git pull自动地识别去哪里抓取。而当需要手动创建的时候，可以使用git checkout -b <branch> <remote>/<branch>。或者直接在检出的时候使用--track。
  • 删除分支：--delete
变基(rebase)

与merge创建一个三方合并不同，变基会提取在某分支上的所有修改，然后移到另一分支上。合并结果并没有区别，但是会使得提交历史更加整洁，这在尤其是在向远端推送的时候非常重要。同样，也可以将a上衍生出的b上衍生出的c中的修改并到a上而不连带着改b中的修改(git rebase --onto a b c)。

但是需要注意的是，变基的实质是丢弃一些现有的提交，相应地新建一些内容一样但实质不同的提交。所以，如果提交存在于自己的仓库以外，可能有别人基于这些提交来开发，就不要变基。否则，将会可能使得别人不得不再次把手上的修改和变基的提交整合，无穷无极。如果真的遇到了，可以使用变基解决变基。

CH5 工作流

集成管理者工作流：每一个人从官方仓库中克隆出来，然后可以提交到自己的仓库，并请求官方仓库的维护者拉取更新合并到主项目。在此基础上，可以分级。

提交原则：
  • 不含空白
  • 单个提交解决单个问题
  • 写好的message，一行摘要+若干行必要的详细解释。

不同模式：

  • 私有项目：具有管理权限，直接以集成管理者模式

  • 共有项目：没有更新权限，fork出一个项目，提交到该仓库，发pull request。

Cmake初步

最初接触cmake的时候是有一点排斥性的，因为对于当时所写的parser项目，我手写的makefile在我看来已经足够好。但在后续源码编译z3求解器的时候深切感受到了cmake的香。

为什么用它

cmake的主要优点，根据wiki的说法，是可以隔离编译产生的输出(compiler outputs)和源(source tree)，从而使得”从源文件产生多个build”和”交叉编译(cross compilation)”成为可能。

^注: 关于交叉编译(corss compiler)，它可以产生与编译器环境不同的平台的可执行代码。它与source-to-source compiler是完全不同的，后者指的是产生目标语言的源代码，而非可执行代码。例如clang就是一个cross-compiler

另一方面，这也让删除产生文件不太会容易影响到源文件（不会删错）。

它能做什么

使用cmake构建项目的工作过程分为两步：

  • 根据配置文件(例如CMakeLists.txt)生成对应系统的构建文件。例如，在Unix上，可能是构建出Makefile
  • 使用对应平台的构建工具实际地编译项目。

如上文所说，cmake被我用来生成makefile。在构建过程中，它会产生一个CMakeCache.txt文件，以及对于每一个指定的子目录产生CmakeFile目录。这会有利于在重复cmake的时候加速。

它怎么用

cmake的语言是较为简单的命令式脚本语言。在源码根目录下会存在一个CMakeLists.txt作为入口。它会指明需要使用的源文件和对应的构建参数。例如：

  • add_executable()会添加可执行文件目标和构建这个目标需要的文件
  • add_library()会添加一个库的目标和构建这个目标需要的文件
  • target_link_libraries()会添加依赖

[SigSEGV]bison - chkstk_ms()

在用bison写parser的时候遇到这个问题，感觉并不是偶然现象，有必要仔细探究一下，判断其对于后续的影响。

问题背景

在语法分析器中，需要暂存当前已移入(shifted)而尚未被规约(not reduced yet)的符号们。举例说明，当解析a+b*c这个表达式，尚未遇到结尾符号c时，会需要暂时保存前面所有的token。因为只有遇到了c，才可以把b*c规约成一个expression，进而和a再规约成更高级的expression。所以在语法分析器设计的时候，为了避免符号栈爆炸，常常会需要仔细考虑表达式和符号的结合性。

在我个人看来，左右结合是各有利弊的。具体来说：

  • 左结合，也即形如((a+b)+c)的优点在于，减少语法分析器中暂存的移入栈中的符号的个数。
  • 右结合，也即形如(a+(b+c))的优点在于，可以在从根节点开始向下访问的时候，首先访问到头结点，并且在每一层都可以实时地知道左节点对右节点的影响。这来自于我的一种直觉：“上文对下文的影响”往往比“下文对上文的影响”多。在一些需要考虑前一个节点是谁来做判断的时候，根据我的经验，这样写更加符合我们的认知习惯。

我们的parser在var list和statement中都采用了右结合。这在设计之初并没有被我们重视。但在输入规模远超预期的时候，它的影响就体现出来了。

问题还原

产生parser的过程并没有任何问题，但在实际parsing的过程中，我们常常会遇到这样的报错：

这个错误并不每次出现，在linux中通常不会出现这个问题，但是在windows下，同样的情况，就有可能会出现这个问题。windows中也并不总是出现这个问题。

最初遇到这个问题的时候，我用valgrind进行内存检查。发现出现了invalid read & write。经排查，发现是bison中未被纳入语法树的节点并不会被自动回收（例如a+b，加号这个token并不会被我并入二叉树，只是在它们产生的父节点中填一个key叫expr_add）。语法树小的时候这微不足道，大的时候这还是很可观的。于是我手动书写了无效节点的回收，然后valgrind检查中就没有了任何错误。但是问题并没有消除。

在cygwin中，确乎是出现了疑似内存访问越界的bug（也即这个SIGSEGV）。逐句调试的时候会发现，其出错发生在调用某函数的一瞬间。（甚至无法达到函数中的第一句语句），就好像是函数压栈的时候出了问题。在进行了很多无效的尝试之后，我修改了bison中的栈大小，从8192改成了16384，就解决了问题。由此分析，好像是栈耗尽产生的问题。

但今天，一个上限设置为163840的parser出现了同样的问题。而我将其改大并没有解决，反倒是改小解决了问题。这就非常让人在意。于是，我去研究了一下这个chkstk_ms()究竟是何方神圣，又是怎么引起了这一次次的错误。

问题溯源

所做的是：接受一个栈大小，以rax传入，然后把它调整到一个4k的倍数，然后以4k为大小向下调整。

借此分析，我们所遇到的问题是，将一个栈的大小传给了chkstk_ms()后，后者访问了该栈的内存页，但引发了崩溃。这种崩溃，根据MSDN的说法，是由GUARD PAGE产生的。在window中，有GUARD PAGE的设定：在分配了若干长度的一个数据结构后，会紧跟着在后面设置一个GUARD PAGE，以避免越界。其利用CPU的异常机制，将PAGE_PRESET位标志为0，访问该页的时候，就会触发一个\#PAGE FAULT 异常，操作系统的事先定义好的处理函数就会被调用。

简而言之，bison的作者认为，操作系统并不一定会在每个页后面（显然，由于虚拟内存的存在，页不一定是连续的）都创造一个guard page，而这会有些危险。所以它设置了，最大的页大小：4032个字节。（这小于linux中普通页的大小4096B，因而在作者看来是安全的）。而当我们把bison的栈的大小开得大于它的时候（也即大于32256），其将可能由于内置的一些运算产生不确定结果。

结论

综上，栈开得小的时候，可能会由于移进栈爆炸而出错，开得过大的时候，则可能出现不稳定的错误。Bison本身不认为会有人一次性处理超过一整个内存页的token，但不幸的是我们写了一个右结合，而输入的用例里，VAR LIST可能确实会很大。

由于改动可能会引起下游的一系列修改（关于这一点的具体工作量存疑），暂时姑且把栈开一个较大值，待后续迫不得已的时候再进行修改。

Parser notes

写了两个parser以后，和之前做math parser时相比有了更多的一些想法。

不得不承认，math parser是一个较为brutal的不完全成品。这种brutal体现在于：

  • 对于节点的结构设计较为随意，没有考虑清楚如何有效利用信息。这使得后续书写二阶key的时候做了不少dirty work，并且破坏了设计结构。
  • 对于结合性的设计较为随意，没有考虑到底层的栈实现。、
  • 对于内存的分配和回收并没有在意，彼时对于这个东西知之甚少。
  • 对于线程安全概念没有了解。
  • 对于跨平台没有什么认知。
  • 对于异常处理和错误恢复也没有考虑
  • 对于链接全然不知。

以下分别简单说说

节点设计

结合性设计

这就值得多说一点了。

在语法分析器中，需要暂存当前已移入(shifted)而尚未被规约(not reduced yet)的符号们。举例说明，当解析a+b*c这个表达式，尚未遇到结尾符号c时，会需要暂时保存前面所有的token。因为只有遇到了c，才可以把b*c规约成一个expression，进而和a再规约成更高级的expression。所以在语法分析器设计的时候，为了避免符号栈爆炸，常常会需要仔细考虑表达式和符号的结合性。

在我个人看来，左右结合是各有利弊的。具体来说：

  • 左结合，也即形如((a+b)+c)的优点在于，减少语法分析器中暂存的移入栈中的符号的个数。
  • 右结合，也即形如(a+(b+c))的优点在于，可以在从根节点开始向下访问的时候，首先访问到头结点，并且在每一层都可以实时地知道前一个节点是谁。这来自于我的一种直觉：“上文对下文的影响”往往比“下文对上文的影响”多。在一些需要考虑前一个节点是谁来做判断的时候，根据我的经验，这样写更加符合我们的认知习惯。

我的parser在list设计中都采用了右结合。这在设计之初并没有被我重视……但在输入规模远超预期的时候，它的影响就体现出来了。由于不读到最后一个就不会规约，右结合这个糟糕的设计让我的符号栈迅速的爆掉了。如果可以的话，还是左结合好一点。

线程安全

这也是之前没有考虑过的点。在多线程调用parser的时候，会因为有符号冲突而出现识别错误。而flex&bison实际上是可以通过设置”pure-parser”来实现线程安全。对于以union作为语法树节点来说，所有的yylval.a都要改成yylval->a，且任何全局变量的定义我认为都可能危险，应该尽力避免。

在要求线程安全的时候，yylex()函数的形式会有一些变化，对于lexer和parser的交互中传递的信息需要手动设置，好在bison中有十分好用的接口，只需要在开头加一些内容就可以了。

跨平台

这部分其实到现在也没有多少认知。我想说的大概是“不要因为代码在自己的设备可以跑就认为它也可以跑在别的设备上”。最初让我意识到这一点的是我把换行符的识别写死成了LF，使得windows上总是报一些奇怪的错误。其次，栈上限（ulimit -s）有时候也会有一些差异，会使得同样的parser在不同环境上的表现大有差别。此外，bison&flex的版本，编译器的版本，也都是需要考虑的问题。这个目前我仍然知之甚浅，需要进一步学习。

异常处理和错误恢复

错误恢复其实是基于这样一种直觉：即使输入了一些能让天塌下来的奇怪的东西，也能找到下一个可信的恢复点。而当读入一个语法错误的时候，就应该直接扔掉接下来的输入，一直到下一个可信恢复点。但是从词法意义上，我认为可信恢复点不一定存在。

关于异常处理我理解的则是，出现错误的东西应该有对应的好的处理办法。最初，我写的是调用yyerror打出error message，但这会让人很难去找到底错在哪里。我之后在每一个token被识别时或者每一次规约的时候设定了#ifdef lexer/parser_DBG来做一些printing work，借助最后一个正确识别的符号来判断出错在哪里。但这同样需要肉眼去搜索。所以后续我借助bison提供的接口追加了location,在报错的时候通过行号来提供更进一步的信息。

而报错之后，同样需要考虑终止程序的事情。之前我写的总是exit()，但是作为动态链接库使用的时候这显然是不可接受的。调用yyterminate()同样会遇到词法和语法分别有一个报错的情况。最终我设置了一个标志位来解决这个问题。

链接中的符号冲突

在同一个程序中调用多个parser的动态链接库的时候，会出现符号冲突。在我们实践过程中，windows本地没有出现问题（存疑）， 但是linux的服务器端出现了问题。关于链接中的符号冲突，我也知之甚少。linux中的nm指令可以扫描动态链接库对外暴露的符号名和类型，这可能回有利于进一步深入这个问题。这里我的解决方案是，把它们的所有yy开头的函数和符号名用宏的方式保护起来。

实际上，bison中是提供了这个功能的，它称之为multi-parser。对于这个问题，bison提供了两种解决方案：

  • 使用条件匹配语法，来让多个文法共存在一个parser里。
  • 使用宏定义保护头。

具体的优劣在其手册中也有表述。而为了不让我写的key的枚举有打架的危险，我选择后一种。在进行了保护以后，多个parser就是相对安全地各自为战不会互相调用了。
